# AGENTS.md — تعليمات وكلاء البرمجة (Codex وغيره) لمشروع village_council_app

> **المرجع الأعلى لطريقة العمل:** `project-brain/PROJECT_CONSTITUTION.md` — اقرأه ولا تكرره هنا.
> هذا الملف يحتوي قواعد المشروع الثابتة. `CLAUDE.md` مرآته لـ Claude Code:
> **أي تعديل على القواعد المشتركة يجب أن يُطبَّق على الملفين معًا حتى لا يتعارضا.**

---

## 1. وصف المشروع

- `village_council_app` — تطبيق Flutter لإدارة **مجالس متعددة**؛ كل مجلس منظمة (organization) مستقلة.
- **Firebase هو مصدر البيانات الحقيقي.**
- التقنيات: Flutter, Dart, Riverpod, go_router, Firebase Auth, Firestore, Storage, FCM, Cloud Functions (Node 20).
- **لا تستخدم Mock أو Dummy أو local-only في ميزات الإنتاج.**

---

## 2. تعدد المجالس (Multi-Tenant)

- **كل عملية مقيّدة بـ `organizationId`.** المسارات تحت `organizations/{organizationId}/…`.
- العمليات الخاصة بالعضو تستخدم `membershipId` و`userId` بصورة صحيحة.
- **يمنع خلط بيانات عضو بين مجلسين.**
- `organizationContextProvider` هو سياق المجلس الحالي في Flutter.
- **لا تستخدم `memberId` وحده في الاستعلامات متعددة المجالس.**
- البيانات المالية والحجوزات والإيصالات والصلاحيات مستقلة لكل مجلس.

---

## 3. Firebase

- افحص Firestore Rules وStorage Rules والفهارس عند إضافة أي مسار جديد.
- **لا تستخدم `allow read, write: if true`.**
- لا تثق ببيانات العميل في العمليات الحساسة؛ العمليات الحساسة **خادمية**.
- استخدم server timestamps.
- استخدم Firestore transactions وidempotency عند الحاجة.
- **لا تنشر Firebase دون إذن صريح.**
- اختبر على Emulator أو staging قبل الإنتاج.
- **لا تنفذ migration على الإنتاج دون نسخة احتياطية وdry-run وموافقة صريحة.**

---

## 4. الأدوار والصلاحيات

- المالك الأعلى `system_owner` لديه `fullAccess`.
- صلاحيات المجلس تأتي من `membership` والدور و`permissionsSnapshot`.
- **لا تعتمد على إخفاء الواجهة كحماية.** تحقق من الصلاحيات في Rules وCloud Functions.
- لا تغيّر الدور أو الصلاحيات دون تدقيق وإشعار عند الحاجة.
- حافظ على استقلال الدور بين المجالس.
- أسماء `roleId` الفعلية الحالية تشمل: `system_owner`، `owner`/`council_owner`،
  `chairman`، `adminManager`، `financialManager`، `financialReviewer`، و`member`.
  `superAdmin` توافق قديم في `platform_admins` وليس دور عضوية جديدًا.

---

## 5. النظام المالي

- **استخدم البيسة كعدد صحيح (`int`) في كل العمليات الجديدة. 1 OMR = 1000 بيسة.**
- اعرض المبلغ **بثلاث خانات عشرية** (`formatBaisa`).
- **لا تستخدم `double` للحسابات الجديدة.**
- كل البيانات المالية مقيّدة بالمجلس والعضوية.
- لا تسمح بقيم سالبة أو دفع زائد.
- التسديد الجزئي يحدّث المدفوع والمتبقي والحالة بصورة صحيحة.
- الاعتماد والرفض والتوزيع عمليات **خادمية وذرية**.
- امنع الاعتماد أو إنشاء الرسم مرتين.
- حافظ على توافق البيانات القديمة حتى اكتمال migration.
- **رفع الإيصال هو وسيلة الدفع الفعّالة حاليًا.**
- `onlinePaymentsEnabled = false` — **لا تُظهر أو تُفعّل الدفع الإلكتروني.**
- **لا تخزّن مفاتيح دفع أو أسرارًا داخل Flutter أو Firestore.**

---

## 6. الدفع عن الآخرين

- يُسمح بالدفع عن النفس أو عضو أو عدة أعضاء **داخل المجلس نفسه**.
- البحث بالاسم داخل الأعضاء النشطين، ويعرض بيانات تعريف محدودة.
- **لا يتم تنزيل قائمة الأعضاء كاملة.**
- مبلغ الإيصال يجب أن **يساوي مجموع `allocations` بالضبط**.
- الخادم **يعيد التحقق** من الرسوم والأرصدة والمستفيدين.
- الإشعارات تصل للدافع والمستفيدين.
- يمنع الدفع الزائد والتكرار والاعتماد المزدوج.

---

## 7. الحجوزات والرسوم

- رسوم `booking`/`event` تُنشأ عند نقطة العمل الصحيحة وبـ **idempotency**.
- إعداد المجلس يحدد: `free` أو `subscription` أو `booking` أو `subscriptionAndBooking`.
- رسوم العضو وغير العضو مستقلة.
- لا تنشئ رسم حجز قبل تحقّق الحالة المطلوبة.
- الإلغاء أو الرفض يجب أن يعالج الرسم بطريقة موثقة.

---

## 8. واجهة المستخدم

- **العربية وRTL هي الأساس.** نصوص عربية سليمة دون ترميز مشوّه.
- تصميم حديث ومتجاوب؛ استخدم مكونات وثيم المشروع.
- وفّر حالات `loading` و`empty` و`error`.
- لا تعرض بيانات مجلس آخر أثناء تبديل السياق.
- راعِ أحجام الهواتف وسهولة الاستخدام.

---

## 9. Git وسلامة الملفات

- **ابدأ كل مهمة بـ `git status`.**
- حافظ على تعديلات المستخدم.
- **لا تستخدم `git reset --hard` أو `checkout` لاسترجاع الملفات دون إذن.**
- لا تحذف ملفات غير متتبعة تخص المستخدم.
- لا تعدّل generated files أو cache بلا حاجة.
- **لا تعمل commit أو push إلا بطلب صريح.**
- لا تخلط تعديلات غير مرتبطة في commit واحد.
- راجع `git diff` قبل التسليم.
- **اترك `.gradle/9.3.0/fileHashes/fileHashes.lock` المحلي كما هو ولا تنظّفه تلقائيًا.**

---

## 10. الأسرار والخصوصية

- لا تضف service-account keys، ولا API keys أو tokens، ولا ملفات `local_keys`.
- لا تضف أرقام حسابات بنكية أو ملفات `xlsx` حسّاسة.
- لا تطبع أسرارًا أو بيانات شخصية في logs.
- استخدم Secret Manager للأسرار الخادمية عند تفعيل أي مزوّد مستقبلي.
- تأكد من `.gitignore`.

---

## 11. الاختبارات وتعريف الاكتمال

قبل اعتبار أي مهمة مكتملة:

```powershell
dart format <الملفات المعدلة>
flutter analyze
flutter test
node --check functions/<الملف المعدل>.js
cd functions; npm test
```

- تحقق من JSON والفهارس.
- اختبر Rules أو Emulator عند تعديل Firebase.
- راجع `git diff`.
- **لا تعتبر المهمة `done` إذا بقيت اختبارات تكامل أو QA أساسية مرتبطة بها.**

## 12. Project Brain

- `project-brain/tasks.json` هو **مصدر حالة المهام**.
- `project-brain/PROJECT_DASHBOARD.md` **ملف مولَّد — لا تعدّله يدويًا.**
- بعد تعديل `tasks.json` شغّل:
  ```powershell
  .\scripts\update-project-dashboard.ps1
  ```
- **نسب الإنجاز يجب أن تكون حقيقية.**
- لا تضع تفاصيل مؤقتة أو يومية داخل `AGENTS.md` أو `CLAUDE.md` — التفاصيل المتغيرة مكانها `project-brain/`.

---

## 13. متى تطلب الإذن

**اطلب الإذن قبل:**

- `firebase deploy`
- migration حقيقي
- تعديل أو حذف بيانات الإنتاج
- `git commit` أو `push`
- حذف ملفات
- أوامر Git المدمّرة
- إضافة مكتبة كبيرة أو تغيير معماري خارج نطاق الطلب
- إدخال مزوّد دفع حقيقي
- التعامل مع مفاتيح أو أسرار

**لا يحتاج إذنًا:**

- قراءة الملفات والبحث داخل المشروع
- تعديل الملفات المطلوبة ضمن المهمة المصرّح بها
- تشغيل formatter وanalyze والاختبارات المحلية غير المدمّرة
- تشغيل dry-run لا يكتب بيانات

## بنية المشروع المختصرة

| المسار | المحتوى |
|---|---|
| `lib/data/models/` | النماذج (`financial_models.dart`, `transaction_model.dart`) |
| `lib/data/repositories/` | الوصول إلى Firestore/Functions |
| `lib/domain/` | منطق خالص قابل للاختبار |
| `lib/presentation/screens/` | الشاشات (`member/`, `admin/`) |
| `lib/providers/app_providers.dart` | مزوّدات Riverpod |
| `lib/router/app_router.dart` | go_router |
| `functions/` | Cloud Functions (`index.js` يصدّر كل شيء، `financial.js`, `audit.js`) |
| `scripts/` | أدوات إدارية وسكربتات النشر والترحيل |
| `project-brain/` | ذاكرة المشروع الرسمية |
| `docs/` | التوثيق المرجعي التفصيلي |

- مشروع Firebase الافتراضي: `alrahmat-console` (`.firebaserc`).
- المنطقة الموحّدة للدوال: `us-central1`.
- **الدوال الجديدة يجب أن تُصدَّر من `functions/index.js` وإلا لن تُنشر.**
