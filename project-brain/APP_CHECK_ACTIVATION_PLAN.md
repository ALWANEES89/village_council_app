# خطة تفعيل Firebase App Check

## النطاق الحساس

يشمل التفعيل التدريجي دوال البحث في دليل الأعضاء، وقراءة الرسوم القابلة للدفع، وإرسال إيصالات العضو أو المستأجر، واعتماد الإيصالات ورفضها، وإدارة الإعدادات والباقات والحسابات والرسوم، وطلبات إلغاء الحجز، وتنظيف الملفات المؤقتة. جميع هذه الدوال تتحقق حاليًا من Firebase Auth ومن المجلس والصلاحية والملكية خادميًا؛ App Check طبقة إضافية وليس بديلًا عن ذلك.

## المتطلبات

- Android: تسجيل تطبيق staging وبصمات SHA المطلوبة ثم Play Integrity. يستخدم debug provider في التطوير فقط.
- iOS: تسجيل تطبيق staging ثم App Attest مع DeviceCheck كخيار احتياطي وفق الأجهزة المدعومة.
- Emulator والتطوير: تمرير debug token من بيئة محلية آمنة إلى Firebase Console الخاصة بـstaging. لا يُحفظ token في Git أو Flutter أو Firestore.
- التأكد من أن كل تطبيق staging يستخدم ملفات إعداد Firebase الخاصة به، لا إعدادات الإنتاج.

## التفعيل التدريجي في staging

1. تفعيل جمع المقاييس دون enforcement ومراقبة الطلبات الشرعية سبعة أيام على الأقل.
2. تسجيل debug tokens لأجهزة QA فقط، والتحقق من Android وiOS وEmulator.
3. تفعيل enforcement أولًا لدوال البحث والقراءة محدودة المخاطر.
4. تفعيله بعد ذلك لإرسال الإيصال ودوال الحجز.
5. تفعيله أخيرًا لدوال الاعتماد والإدارة المالية بعد نجاح سباق الاعتماد والـrollback.
6. لا يشمل scheduled functions؛ هي استدعاءات خادمية وليست callables من التطبيق.

## التراجع

عند ارتفاع أخطاء App Check، يعطّل enforcement للدالة المتأثرة في staging فقط، مع إبقاء Auth والتحقق الخادمي كما هما، ثم تُراجع المقاييس وتهيئة التطبيق قبل إعادة التفعيل. لا تُخفف Firestore Rules أو صلاحيات الدوال كحل مؤقت.

> الحالة الحالية: `enforceAppCheck` غير مفعّل عمدًا، ولم تُضف أي مفاتيح أو tokens.
