# قرار معماري: اعتماد الدفع الإلكتروني تلقائيًا (تحليل — غير منفَّذ)

**التاريخ:** 19 يوليو 2026
**المهمة:** MAJ-8 — تطوير مراجعة الإيصالات والمبالغ
**الحالة:** تحليل ومخطط معتمد للتنفيذ المستقبلي. **لم يُفعَّل الدفع الإلكتروني**، ولم تُغيَّر قاعدة البيانات أو القواعد.
**السبب في عدم التنفيذ الآن:** الدستور و`CLAUDE.md` (بند 5) يفرضان `onlinePaymentsEnabled = false` و«لا تُظهر أو تُفعّل الدفع الإلكتروني». التفعيل الحقيقي يتطلب مزوّد دفع + أسرار في Secret Manager + إذنًا صريحًا (بند 13). لذا هذا القرار **موثَّق وجاهز**، ويُنفَّذ عند صدور الإذن فقط.

---

## 1. الوضع الحالي (حقائق من الشيفرة)

- النظام حاليًا **يدوي فقط**: رفع إيصال (صورة/PDF) ← حالة معلقة ← اعتماد المدير/المراجع المالي.
- علما `FinancialSettings` تحمل علمين: `receiptPaymentsEnabled = true` (يدوي، دائمًا مفعّل)، و`onlinePaymentsEnabled = false` + `onlinePaymentProvider = null`.
- **العلم مثبّت على false في كل مواضع الكتابة**: العميل `financial_models.dart:139-140` (toFirestore يكتب false صراحة)، والخادم `functions/financial.js:750-751` (updateFinancialSettings يفرض false بصرف النظر عن الإدخال). أي لا يمكن للعميل تفعيله إطلاقًا.
- **لا توجد أي شيفرة بوابة دفع**: صفر تطابق لـ`stripe`/`gateway`/`webhook`/أي SDK في `lib/` و`functions/` و`firestore.rules` و`storage.rules`.
- الاعتماد اليدوي يتم داخل `reviewFinancialReceipt` (`financial.js:1368-1482`) في **معاملة Firestore ذرية واحدة**: يتحقق من الرصيد لكل رسم، ثم `amountPaidBaisa += alloc`، `balanceBaisa -= alloc`، والحالة `paid` إذا صار الرصيد صفرًا وإلا `partial`، ويحدّث المعاملة إلى `approved`، ويحذف أقفال pending، ويُدرج في `timeline`.

## 2. القرار المطلوب

نوعان فقط للدفع:

1. **يدوي:** تحويل بنكي + رفع إيصال ← `Pending Review` ← يحتاج اعتماد المدير المالي ← بعد الاعتماد يصبح مدفوعًا. **(هذا هو المطبَّق فعليًا اليوم.)**
2. **إلكتروني:** الدفع داخل التطبيق عبر بوابة موثوقة ← عند وصول Webhook/Callback صحيح والتحقق من التوقيع والعملية ← **اعتماد تلقائي فوري إلى Paid دون المرور على شاشة المدير المالي**.

يتحول دور المدير المالي في الإلكتروني إلى: مراجعة، بحث/تصفية، تدقيق، Refund، Cancel Transaction، متابعة العمليات الفاشلة/المعلقة، تقارير، مراقبة.

## 3. لماذا هذا التصميم صحيح

- **مصدر الثقة يختلف**: في اليدوي، الإثبات صورة يحتاج بشرًا للتحقق. في الإلكتروني، الإثبات توقيع مشفّر من البوابة يتحقق منه الخادم آليًا؛ إضافة اعتماد بشري فوقه تكرار بلا قيمة وتأخير للعضو.
- **الذرية محفوظة**: منطق الاعتماد الموجود (`reviewFinancialReceipt`) هو نفسه المطلوب؛ الفرق الوحيد **مَن يشغّله**: مراجع بشري (يدوي) أو معالج Webhook موثوق (إلكتروني). نعيد استخدام نفس المسار الذري بدل شقّ منطق ثانٍ.
- **عدم الازدواج**: التحقق من التوقيع + idempotency على معرّف عملية البوابة يمنع الاعتماد المزدوج، تمامًا كما تمنع أقفال pending التكرار في اليدوي.

## 4. أثر القرار على كل جزء (تحليل التعارض)

| الجزء | الأثر | التعارض/المعالجة الصحيحة |
|---|---|---|
| **قاعدة البيانات** | إضافة `paymentMethod: manual\|electronic` على المعاملة، و`gatewayRef`/`gatewayEventId` (فريد للـidempotency)، وحقول Refund. **لا تغيير على تخزين البيسة كعدد صحيح.** | لا تعارض. حقول جديدة اختيارية؛ البيانات القديمة تبقى `manual` ضمنيًا. |
| **مسار الاعتماد** | معالج Webhook خادمي جديد (HTTPS onRequest) يتحقق من التوقيع ثم يستدعي **نفس منطق** `reviewFinancialReceipt` (approve) داخل معاملة. | تعارض: التوقيع يحتاج سرًّا. **المعالجة:** Secret Manager فقط (بند 10)، ولا مفتاح داخل Flutter/Firestore. |
| **Audit Logs** | فعل جديد `receipt.auto_approved` (أو `payment.electronic.captured`) بمصدر `payment_webhook` وممثّل النظام لا مستخدم بشري. | يجب إضافة تسمية عربية له في `audit_logs_screen.dart:627-660`، وإلا ظهر خامًا. **موثّق كبند تنفيذ.** |
| **Membership/Booking/Event Fees** | لا فرق في نوع الرسم؛ الإلكتروني مجرد وسيلة سداد لأي `chargeType`. رسوم الحجز الإلكترونية تعتمد فور نجاح الدفع بدل انتظار المراجع. | لا تعارض بنيوي. تنبيه: `event` ما زال بلا دورة إنشاء إنتاجية (config فقط)، فلا يُفعَّل دفعه الإلكتروني قبل بناء دورته. |
| **Financial Dashboard/Reports** | تحتاج تمييز «مدفوع إلكترونيًا» عن «مدفوع يدويًا» + طابور «فاشلة/معلقة إلكترونيًا» لدور المراجعة الجديد. | لا تعارض؛ إضافة تصفية على `paymentMethod` و`Failed`. |
| **Notifications** | نجاح الدفع الإلكتروني ← إشعار «تم الدفع» فورًا. **لا يُلمس نظام الإشعارات القائم**؛ يُعاد استخدام outbox الحالي بقالب جديد فقط. | ضمن حدود العمل: لا تعديل بنية الإشعارات. |
| **صلاحيات المدير المالي** | تبقى صلاحيات الاعتماد اليدوي كما هي. تُضاف مفاهيم `payments.refund`/`transactions.cancel` لدور الإلكتروني. | يجب تحديث `hasNoPrivilegedPermissions` (`firestore.rules:180-202`) وبذور الأدوار لمنع منح العضو هذه الصلاحيات لنفسه. |
| **Organization Bootstrap** | يبقى `onlinePaymentsEnabled=false` افتراضيًا لكل مجلس جديد. لا تفعيل تلقائي. | لا تعارض؛ التفعيل قرار إداري صريح لكل مجلس. |
| **Idempotency/الأمان** | مفتاح فريد على `gatewayEventId`؛ رفض إعادة التشغيل؛ التحقق من العملة (OMR) والمبلغ (بيسة int) ومطابقتهما للرسوم. | لا تعارض؛ يوسّع نمط idempotency الحالي. |

## 5. الخلاصة

- التصميم **متوافق** مع المعمارية الحالية ولا يتطلب هدمها؛ إعادة استخدام مسار الاعتماد الذري + outbox + audit.
- نقاط التوسعة الثلاث قبل أي تفعيل: (1) معالج Webhook موقّع في Secret Manager، (2) حقول `paymentMethod`/`gatewayEventId`/Refund + idempotency، (3) تسميات Audit وصلاحيات Refund/Cancel وتصفية Dashboard.
- **لم يُنفَّذ أي من ذلك في هذه الجلسة** التزامًا بالدستور. يُنفَّذ عند إذن صريح + توفر مزوّد دفع + بيئة اختبار مصرّح بها فقط.

- **الحكم النهائي: MAJ-8 جاهزة بنسبة 100%، و`BUG-002` مغلق بتحقق الجهاز. أُغلقت `TASK-004` بنسبة 100%.** تبقى App Check وiOS والنشر والدفع الإلكتروني الحقيقي أعمالًا لاحقة مستقلة، ولم تُنفذ في هذه الجولة.
