# POST_DEPLOY_CHECKLIST — بعد النشر

> تحقّق فوري من صحّة النشر على البيئة المستهدفة، مع مراقبة أوّلية.

## 1. تأكيد النشر
- [ ] Firebase Console → Functions: كل الدوال الـ 10 ظاهرة وحالتها Healthy (2 إشعارات + 8 audit).
- [ ] Console → Firestore → Rules: النسخة المنشورة هي الأحدث (تاريخ/وقت مطابق).
- [ ] Console → Storage → Rules: مطابقة (إن نُشرت).
- [ ] إصدار التطبيق المنشور هو الذي يمرّر `actorUserId` ولا يكتب audit من العميل.

## 2. اختبار دخان (Smoke Test) على الإنتاج
- [ ] تسجيل دخول حساب حقيقي ينجح (لم يُكسر الدخول).
- [ ] عملية حسّاسة واحدة (مثل اعتماد إيصال تجريبي) → تظهر في Audit Viewer خلال ثوانٍ.
- [ ] `actorName`/`actorRole` صحيحان و`source=cloud_function`.
- [ ] Audit Viewer يعرض مجلس المستخدم فقط (لغير superAdmin).

## 3. تحقّق العزل السريع
- [ ] بحساب مدير مجلس، لا تظهر مجالس أخرى في سجل الأحداث.
- [ ] لا سجلات مجلس آخر تظهر.

## 4. مراقبة أوّل 24–48 ساعة
- [ ] Functions Logs: لا ارتفاع في نسبة الفشل، لا حلقات، لا أخطاء إسناد فاعل متكررة.
- [ ] لا شكاوى "Permission denied" لمستخدمين شرعيين (قد تعني قاعدة أضيق من اللازم).
- [ ] استهلاك القراءات/الكتابات ضمن المتوقّع (السجل يضيف كتابات).
- [ ] لا سجلات audit مكرّرة (idempotency عبر event.id يعمل).

## 5. عند اكتشاف خلل
- [ ] خطأ في القواعد يمنع مستخدمين شرعيين → **Rollback** فوري (Console → Rules History) + تحقيق.
- [ ] دالة audit تفشل باستمرار → راجع الحقول المتوقّعة؛ حذف/إعادة نشر عند اللزوم.
- [ ] وثّق أي حادث وقراره في `CHANGELOG.md` (وربما `SYSTEM_DECISIONS.md`).

## 6. توثيق
- [ ] تحديث `CHANGELOG.md` بالإصدار المنشور وتاريخه.
- [ ] إغلاق بنود `PRE_DEPLOY_TEST_CHECKLIST.md` المنفّذة.
