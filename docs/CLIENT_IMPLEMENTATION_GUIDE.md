# CLIENT_IMPLEMENTATION_GUIDE — دليل تطبيق العميل (تهيئة مجلس جديد)

> دليل تسليم/تشغيل مجلس على المنصّة. المنصّة منتَج Multi-Tenant يُباع لعدّة مجالس؛ هذا الدليل يبقى **جاهزاً للتسليم** ويُحدَّث مع كل ميزة.

## 1. نموذج التشغيل
- منصّة واحدة (Firebase Project واحد) تخدم كل العملاء (المجالس).
- كل مجلس = مؤسسة معزولة (`organizations/{organizationId}`) ببياناته وأعضائه وصلاحياته وسجلّه — **لا يرى مجلس بيانات مجلس آخر إطلاقاً**.

## 2. خطوات تهيئة مجلس جديد (Onboarding)
1. **إنشاء المجلس:** يقوم **سوبر أدمن المنصّة** بإنشاء المجلس (يولّد `organizationId` ويبذر الأدوار/الإعدادات/الملف المالي تلقائياً).
2. **تعيين رئيس المجلس (chairman):** المسؤول الأول للعميل — يملك `fullAccess` داخل مجلسه فقط.
3. **ضبط الإعدادات:** اسم المجلس، اللغة، المنطقة الزمنية، العملة، تفعيل تأجير القاعة (إن رغب).
4. **الملف المالي:** إدخال بيانات البنك وتفعيلها لاستقبال الإيصالات.
5. **الأدوار:** مراجعة/تخصيص صلاحيات الأدوار (مالي/إداري/أمين سر) حسب حاجة المجلس.
6. **دعوة الأعضاء:** عبر رابط/QR انضمام؛ الأعضاء يقدّمون طلبات تُعتمد من الإدارة.

## 3. ضمانات العزل للعميل (Data Isolation Guarantees)
- بيانات المجلس (أعضاء، مدفوعات، حجوزات، سجل) معزولة بالمسار والقواعد.
- لا مدير أو عضو يصل لمجلس آخر.
- سجل الأحداث معزول لكل مجلس ولا يُعدَّل/يُحذف (append-only، خادمي).
- الصلاحيات محصورة بالمجلس الحالي.

## 4. Checklist الإطلاق (Go-Live) لكل مجلس
- [ ] المجلس مُنشأ بمعرّف خاص (لا اسم ثابت).
- [ ] رئيس المجلس مُعيّن ويستطيع الدخول.
- [ ] الإعدادات + بيانات البنك مضبوطة ومفعّلة.
- [ ] الأدوار مراجَعة ومطابقة لحاجة المجلس.
- [ ] اختبار: طلب انضمام → اعتماد → رفع إيصال → اعتماد → ظهور في سجل الأحداث.
- [ ] اختبار العزل: مدير هذا المجلس لا يرى مجلساً آخر.
- [ ] الإشعارات تعمل على أجهزة العميل.

## 5. التوسّع (Scale)
- إضافة المجلس رقم 10 أو 100 لا تتطلب أي تغيير في الكود أو القواعد — بيانات جديدة فقط.
- المنصّة تدعم آلاف المستخدمين وملايين السجلات دون إعادة تصميم.
